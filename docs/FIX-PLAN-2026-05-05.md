# bodycomp 数据连接整改方案

> 日期：2026-05-05
> 问题：用户-教练消息不通、绑定假成功、硬编码假数据泛滥

---

## 一、根因诊断：ID 链条断裂

整个系统的核心问题是一条**断裂的 ID 链**:

```
登录ID ≠ coaches._id ≠ coach_members.coach_id ≠ 会员端查询ID
```

| 环节 | 用的 ID | 从哪里来 | 问题 |
|------|---------|----------|------|
| 登录（finishLogin） | `user.id = uid 或邀请码` | CloudBase UID | 存入 localStorage |
| 绑定教练（bindCoach） | `coach._id` | `coaches` 集合的文档 _id | ✅ 正确 |
| 教练查会员（getMyMembers） | `coachId = user.id` | localStorage `user.id` | ❌ 错误！`user.id` ≠ `coaches._id` |
| 教练发消息 | `memberId = 路由参数` | URL `/coach/members/:id` | ❌ 路由 ID 可能不带前缀 |
| 会员查消息（getNotifications） | `memberId = 'MB202604001'` | 硬编码默认值 | ❌ 与 CloudBase user_id 不匹配 |

## 二、问题清单（5 类，13 个具体项）

### 🔴 P0：绑定是假的（最严重）

**问题 1**：`MemberProfilePage.tsx` 的 `handleBindCoach()` 仅操作本地 state，从未调用 CloudBase API

```tsx
// 当前代码（MemberProfilePage.tsx 218-225）
const handleBindCoach = () => {
  const coach = availableCoaches.find(c => c.id === selectedCoachId)
  if (coach) {
    setCoach(coach)      // ← 仅本地
    setHasCoach(true)    // ← 仅本地
    close()
  }
}
```

**修复**：需要在绑定回调中同时调用 CloudBase 的 `bindCoach()` API，并同步教练端可查询到的数据。

---

**问题 2**：`coachApi.getMyMembers()` 查询用错 ID，导致教练永远查不到真实会员

```tsx
// 当前代码（api.ts 489-496 简化逻辑）
coachApi.getMyMembers = async (coachId) => {
  const cid = coachId || getCurrentUserId()  // ← 从 localStorage 读 user.id
  // 查 coach_members.where({ coach_id: cid })
  // 但 cid = CloudBase UID，而 coach_id = coaches 集合的 _id
}
```

**修复**：教练端查会员前，需先通过 `user.id` 找到对应 coach 记录，取其 `_id`，再用 `_id` 查 `coach_members`。

---

### 🔴 P0：消息不通

**问题 3**：教练端发消息只写本地 `notificationStore`，会员端用硬编码 ID 查不到

```tsx
// CoachMemberDetailPage.tsx 302-319
handleSendMessage → addNotification({ memberId: memberId || undefined })
// 只写 zustand store，未写 CloudBase 数据库

// MemberMessagesPage.tsx 84
const memberId = profile.memberId || 'demo_member'  // ← 永远是 'MB202604001'
```

**修复**：
- 发送消息需同时写 CloudBase `notifications` 集合
- 会员端读取需用 CloudBase 真实 user_id

## 三、修复方案（5 步，按依赖顺序）

### 第一步：修复 ID 链 — 教练查会员

**目标**：让 `coachApi.getMyMembers()` 用正确的 coach._id 查询

**改动文件**：`services/api.ts` — `getMyMembers()` 函数

**逻辑**：
1. 从 localStorage 获取 `user.id`（CloudBase UID）
2. 用 `user.id` 查 `coaches` 集合（`coaches.where({ user_id: user.id })`），拿到 coach._id
3. 用 coach._id 查 `coach_members`（`coach_members.where({ coach_id: coach._id })`）
4. 拿到 member_id 列表，批量查 users + body_records + assigned_plans
5. 返回真实数据（查不到返回 `[]`, 不 fallback 到假数据）

---

### 第二步：修复绑定 — 真正调用 CloudBase

**目标**：绑定操作同时写 CloudBase 数据库

**改动文件**：
- `MemberProfilePage.tsx` — `handleBindCoach()` 函数
- `cloudbase/services/coach.ts` — `bindCoach()` 可能需要调整

**逻辑**：
1. 绑定前检查：当前 CloudBase 用户 ID（`getCurrentUser()`）
2. 调用 `bindCoach(coach._id)` 写入 `coach_members` 集合
3. 同时更新 `coaches` 集合的 `member_count`
4. 成功后更新本地 state

---

### 第三步：修复消息 — 教练↔会员双向打通

**目标**：教练发消息 → 会员收到；会员回复 → 教练看到

**改动文件**：
- `CoachMemberDetailPage.tsx` — `handleSendMessage()`
- `MemberMessagesPage.tsx` — 消息查询
- `MemberProfilePage.tsx` — 消息入口

**逻辑**：
1. 教练发消息：同时写本地 store + CloudBase `notifications` 集合（`user_id = 会员的CloudBase UID`）
2. 会员查消息：用 CloudBase 真实 user_id（非 `'MB202604001'`）
3. `cloudbase/services/notifications.ts` 已有 `sendNotification()` 和 `getNotifications()`，直接调用即可

---

## 四、实施顺序（重要）

```
第一步 → 第二步（绑定依赖 ID 链修复）
    ↓
第三步（消息依赖绑定完成后 memberId 正确）
```
